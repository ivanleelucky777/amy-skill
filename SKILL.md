# Amy Skill｜Amy 親子 KOL 多平台內容產製系統

> **本檔性質**：這套 skill 的「入口外殼」。實質規則分散在同資料夾的多個模組檔，本檔只負責「啟動指引 + 指出哪份檔放什麼」，**不複述規則內容**（守單一真相源）。

## 🚦 啟動：第一步永遠先讀 `router.md`

收到任何「為 Amy 產製／調整內容」的訊號時，**先讀 `router.md`，不要跳過**。

`router.md` 是這套系統的神經中樞——啟動 SOP、對話分流、規則索引、衝突清單、決策節點全在那裡。先讀它的理由很實際：

- 它會告訴你**這次對話該走哪條路徑**（接案產製／修文案／Skill 維護／狀態監測），以及**該載入哪幾份模組**（按需載入，不是一次全讀）。
- 它有一份「**衝突清單**」記著哪些舊規則已被推翻。**跳過這步最常見的災難，就是引用到 `archive/` 裡的舊版規則去產製，寫出整包錯的內容。** 先讀 router 才不會踩到。

## 🔴 三條不可妥協的紅線（內容在別處，本檔只標位置）

下面三條是**最高優先、凌駕一切**的鐵則。它們的完整內容在 `principles.md` PART I，這裡只點名提醒，不複述：

1. **體驗真實性分流** — Amy 沒完整體驗的活動只能用「告知」語氣，不得用實吃口吻腦補。
2. **不腦補（事實核對）** — 店家品類、地理位置、工法、份量、人名……素材沒寫的不准自編。
3. **演算法優先 + Ivan 指令例外** — 預設以演算法最佳化為主；廠商特殊要求只在 Ivan 明確指令時才採用。

詞彙層的紅線在 `dictionary.md` 的 🔴 區（立即替換）。

**這幾條為什麼維持強制語氣、不軟化**：它們對應的是真實風險——造假傷害信任、廣告法觸法、踩中帳號健康度地雷。其餘寫作原則可以用「理解理由後自行判斷」的彈性處理，但這幾條不行，違反就退回重寫。

## 📌 STATE：即時讀，絕不烘進交付物

`state/` 底下三份是**動態狀態**，不是規則：

- `state/account_health.md`（帳號健康度週快照）
- `state/active_constraints.md`（當前限制、廠商例外、Ivan 例外指令）
- `state/performance_log.md`（案件成效歸因）

讀法兩條：

1. **產製／決策前讀它們**（例如：這週是否處於暫停重度業配的限制期？略過率在哪個燈號？）。
2. **絕對不把 STATE 內容寫進任何交付物或交付包骨架。** STATE 會隨時間和帳號變動，烘進去就會過時、污染後續案件。

這條「靜態規則／動態狀態分離」是整套架構的地基，**也是未來要接自動化時能順利分層的關鍵**——skill 包出規則，state 在執行時即時讀。

## ⚙️ 產製流程一眼版

收到案件後的主幹（完整步驟在 `router.md` SECTION B 與 `workflows.md` PART I）：

```
事實核對 → 規劃六區交付包 → Hook/標題提案（給選不寫死）
        → 產製六區內容 → 全平台清點 → 發稿前完整檢查
        → 自動套交付包裝層輸出「可複製 HTML 頁」（DLV-PKG-H5）
```

六區交付包用**永久 ID**（DLV-1～DLV-6），不以編號口頭指稱。規格的唯一 owner 是 `workflows.md`，其餘檔案只用 ID 指標指過去，不重複定義。

## 🧭 衝突與治理（簡版，細節在 router／changelog）

- **新版自動贏舊版**：日期較新的規則永遠優先。
- **不引用歸檔**：`archive/` 內容只供 Ivan 回看歷史脈絡，產製時不採用。
- **單一真相源**：一個事實只有一個 owner 檔，其他檔用 ID 指標引用，不複製內容。
- 看到舊檔提到某規則 → 先去 `router.md` SECTION F「衝突清單」確認它是否已被推翻。
